# Requirements Document

## Introduction

This document defines requirements for the **JobAgent AI Platform Enhancements** — a set of 10 new features and capability upgrades added to the production-ready JobAgent AI platform (Phases 1–7 complete). The enhancements span five new AI agents/capabilities and five upgrades to existing features, all integrated into the existing LangGraph agent harness, FastAPI backend, Next.js 14 frontend, and Supabase/pgvector data layer.

All new agents follow the established conventions: registered in `harness.py` and `orchestrator.py`, routes added under `/api/v1/`, model calls routed through `model_router.py`, actions logged to `agent_runs`, and human-in-the-loop gates enforced server-side before any outbound action.

---

## Glossary

- **Cover_Letter_Agent**: New LangGraph agent responsible for generating personalized cover letters using RAG context.
- **Interview_Coach_Agent**: New LangGraph agent that conducts mock interview sessions, evaluates answers, and provides scoring.
- **Salary_Agent**: New LangGraph agent that retrieves real-time salary data and generates negotiation scripts.
- **Company_Research_Agent**: New LangGraph agent that researches target companies and stores intel in pgvector.
- **NL_Search_Agent**: New LangGraph agent that interprets natural language job search queries and translates them to structured job board queries.
- **Cover_Letter**: A personalized document generated by the Cover_Letter_Agent for a specific job application.
- **Interview_Session**: A stateful mock interview interaction managed by the Interview_Coach_Agent.
- **Salary_Report**: A structured document containing market salary benchmarks and a negotiation script for a specific role and company.
- **Company_Intel**: Structured research data about a target company stored in the `{user_id}_company` pgvector collection.
- **Resume_Persona**: A named resume profile (e.g. "Backend Engineer", "Tech Lead") maintained by a user in the Multi-Resume Profile Manager.
- **ATS_Score**: A numeric score (0–100) representing how well a tailored resume matches a job description for ATS keyword coverage, readability, and format compliance.
- **Kanban_Board**: The existing `/applications` page displaying job applications as cards across status columns.
- **Activity_Timeline**: A chronological log of all events (status changes, emails, agent actions) for a single job application.
- **Follow_Up_Agent**: The existing LangGraph agent that schedules and sends follow-up emails; enhanced in Requirement 7.
- **RAG_Agent**: The existing LangGraph agent that retrieves context from pgvector collections.
- **Orchestrator**: The existing LangGraph supervisor agent that routes tasks to sub-agents.
- **Model_Router**: The `app/core/model_router.py` module that routes all LLM calls through the user's active `user_model_settings`.
- **HITL_Gate**: Human-in-the-loop approval gate enforced server-side via the `/approve` endpoint before any outbound action.
- **Exa_Service**: The Exa.ai REST API integration used for semantic web search.
- **Firecrawl_Service**: The Firecrawl REST API integration used for clean web scraping.
- **Proxycurl_Service**: The Proxycurl REST API integration used for LinkedIn profile data retrieval.
- **PinchTab**: The browser automation service (port 9867) used for job board interaction.
- **BullMQ_Worker**: The Node.js BullMQ worker that processes background agent jobs.
- **pgvector**: The PostgreSQL vector extension used for RAG storage and retrieval.
- **AES_Encryption**: AES-256-GCM encryption applied to all user API keys at rest.

## Requirements

### Requirement 1: AI Cover Letter Generator

**User Story:** As a job seeker, I want a dedicated AI agent to generate personalized, job-specific cover letters using my resume and the job description, so that each application has a compelling, tailored cover letter without manual writing effort.

#### Acceptance Criteria

1. WHEN a user requests a cover letter for a job application, THE Cover_Letter_Agent SHALL retrieve the top 5 resume chunks and the full job description text from pgvector using the RAG_Agent before generating output.
2. WHEN generating a cover letter, THE Cover_Letter_Agent SHALL accept a tone parameter with exactly three valid values: `formal`, `casual`, and `bold`, and apply the selected tone consistently throughout the generated text.
3. IF a tone parameter is not provided by the user, THEN THE Cover_Letter_Agent SHALL default to `formal` tone.
4. WHEN a cover letter is successfully generated, THE Cover_Letter_Agent SHALL store the result in the `user_documents` table with `doc_type = 'cover_letter'` and associate it with the corresponding `job_applications` record via `cover_letter_id`.
5. THE Cover_Letter_Agent SHALL maintain a version history of all cover letters generated for a single job application, retaining a minimum of 10 versions per application.
6. WHEN a user views the cover letter history for an application, THE Cover_Letter_Agent SHALL return all stored versions ordered by `created_at` descending, including the tone used for each version.
7. WHEN a cover letter generation run completes, THE Cover_Letter_Agent SHALL log the run to the `agent_runs` table with `agent_type = 'cover_letter'`, including `tokens_used` and `duration_ms`.
8. THE Cover_Letter_Agent SHALL route all LLM calls through the Model_Router using the user's active `user_model_settings`.
9. WHEN a cover letter is generated, THE System SHALL present the output to the user for review before associating it with an application submission, enforcing the HITL_Gate.
10. THE Cover_Letter_Agent SHALL expose its functionality via a new API route at `POST /api/v1/cover-letter/generate`.
11. WHEN a user requests cover letter generation and no primary resume exists in `user_documents`, THEN THE Cover_Letter_Agent SHALL return an error response with HTTP status 422 and a message indicating that a primary resume must be uploaded first.

### Requirement 2: AI Mock Interview Coach

**User Story:** As a job seeker, I want an interactive AI interview coach that generates role-specific questions, evaluates my answers in real-time, and gives me improvement tips, so that I can practice and improve my interview performance before the real interview.

#### Acceptance Criteria

1. WHEN a user starts a mock interview session, THE Interview_Coach_Agent SHALL generate a set of interview questions tailored to the target role and job description, covering at least one question from each of the three supported types: `behavioral`, `technical`, and `situational`.
2. THE Interview_Coach_Agent SHALL accept a `question_type` filter parameter that restricts generated questions to a single type: `behavioral`, `technical`, or `situational`; when omitted, all three types are included.
3. WHEN a user submits an answer to an interview question, THE Interview_Coach_Agent SHALL evaluate the answer and return a score between 0 and 100, a qualitative rating label (`poor`, `fair`, `good`, `excellent`), and at least one specific improvement tip within 10 seconds of answer submission.
4. THE Interview_Coach_Agent SHALL maintain session state across multiple question-answer turns within a single Interview_Session using LangGraph's stateful graph execution.
5. WHEN an Interview_Session is completed (all questions answered or user ends session), THE Interview_Coach_Agent SHALL generate a session summary containing the overall score (average of individual answer scores), a list of strengths, and a list of areas for improvement.
6. THE Interview_Coach_Agent SHALL store each completed Interview_Session in a new `interview_sessions` table, including the role, question set, answers, scores, and session summary.
7. WHEN a user requests a new interview session for a role, THE Interview_Coach_Agent SHALL retrieve relevant company intel from the `{user_id}_company` pgvector collection if Company_Intel exists for the target company, and incorporate it into question generation.
8. WHEN an Interview_Session run completes, THE Interview_Coach_Agent SHALL log the run to the `agent_runs` table with `agent_type = 'interview_coach'`, including `tokens_used` and `duration_ms`.
9. THE Interview_Coach_Agent SHALL route all LLM calls through the Model_Router using the user's active `user_model_settings`.
10. THE Interview_Coach_Agent SHALL expose session management via new API routes: `POST /api/v1/interview/session/start`, `POST /api/v1/interview/session/{session_id}/answer`, and `GET /api/v1/interview/session/{session_id}/summary`.
11. IF a user submits an answer that is fewer than 10 words, THEN THE Interview_Coach_Agent SHALL return a validation error with HTTP status 422 and prompt the user to provide a more complete response.

### Requirement 3: Salary Intelligence & Negotiation Assistant

**User Story:** As a job seeker, I want an AI agent that pulls real-time salary data, benchmarks my offer against market rates, and generates a word-for-word negotiation script, so that I can negotiate confidently and maximize my compensation.

#### Acceptance Criteria

1. WHEN a user requests a salary report for a role and company, THE Salary_Agent SHALL query the Exa_Service for current salary data using the role title, company name, and location as search parameters.
2. THE Salary_Agent SHALL return a Salary_Report containing: the 25th, 50th, and 75th percentile salary figures for the role and location, the data sources used, and the retrieval timestamp.
3. WHEN a user provides their current offer amount, THE Salary_Agent SHALL compare the offer against the retrieved market percentiles and classify it as `below_market` (below 25th percentile), `at_market` (25th–75th percentile), or `above_market` (above 75th percentile).
4. WHEN a Salary_Report is generated, THE Salary_Agent SHALL produce a negotiation script tailored to the specific role, company, and offer classification, containing at minimum: an opening statement, a counter-offer amount (set at the 75th percentile), and two supporting justification points.
5. THE Salary_Agent SHALL store each generated Salary_Report in a new `salary_reports` table associated with the user and optionally linked to a `job_applications` record.
6. WHEN a Salary_Agent run completes, THE Salary_Agent SHALL log the run to the `agent_runs` table with `agent_type = 'salary_intelligence'`, including `tokens_used` and `duration_ms`.
7. THE Salary_Agent SHALL route all LLM calls through the Model_Router using the user's active `user_model_settings`.
8. IF the Exa_Service returns no salary data for the requested role and location, THEN THE Salary_Agent SHALL return a partial Salary_Report with a `data_unavailable` flag set to `true` and omit the negotiation script, returning HTTP status 206.
9. THE Salary_Agent SHALL expose its functionality via new API routes: `POST /api/v1/salary/report` and `GET /api/v1/salary/report/{report_id}`.
10. THE System SHALL present the generated negotiation script to the user for review before any outbound use, enforcing the HITL_Gate.

### Requirement 4: Company Research Agent

**User Story:** As a job seeker, I want an AI agent that automatically researches target companies and stores the intel for use across all my agents, so that my resume, cover letter, and interview prep are all informed by up-to-date company context.

#### Acceptance Criteria

1. WHEN a user triggers company research for a target company, THE Company_Research_Agent SHALL retrieve data from at minimum four sources: company website (via Firecrawl_Service), recent news (via Exa_Service), tech stack information (via Exa_Service), and Glassdoor sentiment (via Exa_Service).
2. THE Company_Research_Agent SHALL structure the retrieved data into a Company_Intel object containing: company overview, culture summary, recent news items (minimum 3), tech stack list, and Glassdoor sentiment score (positive/neutral/negative).
3. WHEN Company_Intel is successfully compiled, THE Company_Research_Agent SHALL embed and store it in the `{user_id}_company` pgvector collection, namespaced per user, so that the RAG_Agent can retrieve it for other agents.
4. THE Company_Research_Agent SHALL also persist a structured JSON snapshot of the Company_Intel in a new `company_intel` table, associated with the user and company name, with a `researched_at` timestamp.
5. WHEN the Cover_Letter_Agent, Resume_Agent, or Follow_Up_Agent requests context for a company that has existing Company_Intel, THE RAG_Agent SHALL retrieve the top 3 relevant Company_Intel chunks from the `{user_id}_company` collection and include them in the agent's context.
6. WHEN a Company_Research_Agent run completes, THE Company_Research_Agent SHALL log the run to the `agent_runs` table with `agent_type = 'company_research'`, including `tokens_used` and `duration_ms`.
7. THE Company_Research_Agent SHALL route all LLM calls through the Model_Router using the user's active `user_model_settings`.
8. IF the Firecrawl_Service or Exa_Service returns an error for a specific source, THEN THE Company_Research_Agent SHALL continue research using the remaining available sources and include a `partial_data` flag in the Company_Intel object indicating which sources failed.
9. THE Company_Research_Agent SHALL expose its functionality via new API routes: `POST /api/v1/company/research` and `GET /api/v1/company/{company_name}/intel`.
10. WHEN Company_Intel already exists for a company and is less than 7 days old, THE Company_Research_Agent SHALL return the cached Company_Intel without re-fetching, unless the user explicitly requests a refresh.

### Requirement 5: Natural Language Job Search

**User Story:** As a job seeker, I want to describe the job I'm looking for in plain language instead of filling out keyword filters, so that I can find relevant roles faster and more intuitively.

#### Acceptance Criteria

1. WHEN a user submits a natural language job search query, THE NL_Search_Agent SHALL parse the query and extract structured search parameters including: role title, seniority level, location/remote preference, industry or company type, and any additional constraints stated by the user.
2. THE NL_Search_Agent SHALL translate the extracted parameters into a structured job board query and pass it to the existing Job_Search_Agent for execution via PinchTab.
3. WHEN the NL_Search_Agent parses a query, THE NL_Search_Agent SHALL return a structured interpretation summary to the user showing the extracted parameters before executing the search, allowing the user to confirm or correct the interpretation.
4. IF the NL_Search_Agent cannot extract a role title from the query, THEN THE NL_Search_Agent SHALL return an error response with HTTP status 422 and prompt the user to include a job title or role type in their query.
5. THE NL_Search_Agent SHALL support queries containing compound preferences, such as remote work, salary range, company size, and technology stack, and include all extractable parameters in the structured query.
6. WHEN a natural language search is executed, THE NL_Search_Agent SHALL log the original query text, the extracted parameters, and the number of results returned to the `agent_runs` table with `agent_type = 'nl_job_search'`.
7. THE NL_Search_Agent SHALL route all LLM calls through the Model_Router using the user's active `user_model_settings`.
8. THE NL_Search_Agent SHALL expose its functionality via a new API route at `POST /api/v1/jobs/search/natural`.
9. WHEN a natural language search returns results, THE System SHALL display the structured interpretation summary alongside the results so the user can understand how their query was interpreted.

### Requirement 6: Enhanced Application Kanban

**User Story:** As a job seeker, I want an upgraded kanban board with drag-and-drop status management, a per-application activity timeline, AI-suggested next actions, and linked email threads, so that I can manage my job pipeline more effectively from a single view.

#### Acceptance Criteria

1. THE Kanban_Board SHALL support drag-and-drop reordering of application cards between status columns (`saved`, `applied`, `viewed`, `interview`, `offer`, `rejected`) without requiring a page reload.
2. WHEN a user drags an application card to a new status column, THE System SHALL update the `job_applications.status` field in the database within 500ms and display a confirmation indicator on the card.
3. WHEN a user opens an application detail view, THE System SHALL display an Activity_Timeline showing all events for that application in chronological order, including: status changes (with timestamp and actor), agent actions (from `agent_runs`), and email events (sent/received).
4. THE System SHALL display at least one AI-suggested next action on each application card, generated by the Orchestrator based on the current application status, days since last activity, and available Company_Intel.
5. WHEN an application has associated email threads in Gmail, THE System SHALL display a linked email thread count on the application card and allow the user to expand and read the thread inline within the application detail view.
6. WHEN a user moves an application to `interview` status, THE System SHALL automatically trigger the Orchestrator to suggest scheduling the Interview_Coach_Agent for that role.
7. WHEN a user moves an application to `offer` status, THE System SHALL automatically trigger the Orchestrator to suggest running the Salary_Agent for that role and company.
8. THE Kanban_Board SHALL support filtering cards by status, company name, and date range without a page reload.
9. WHEN the Kanban_Board loads, THE System SHALL fetch and display all applications within 2 seconds for a user with up to 100 active applications.
10. THE System SHALL persist the user's column sort order preference in `user_preferences` across sessions.

### Requirement 7: Smart Follow-Up Personalization

**User Story:** As a job seeker, I want my follow-up emails to be personalized using company research and the specific job context rather than generic templates, so that my follow-ups stand out and feel authentic to the recipient.

#### Acceptance Criteria

1. WHEN the Follow_Up_Agent generates a follow-up email, THE Follow_Up_Agent SHALL query the RAG_Agent for Company_Intel from the `{user_id}_company` pgvector collection for the target company before composing the email.
2. WHEN Company_Intel is available for the target company, THE Follow_Up_Agent SHALL incorporate at least one company-specific detail (recent news, culture reference, or tech stack mention) into the follow-up email body.
3. WHEN the Follow_Up_Agent generates a follow-up email, THE Follow_Up_Agent SHALL also retrieve the original job description text and the tailored resume used for the application from `job_applications` and `user_documents` to inform the email content.
4. THE Follow_Up_Agent SHALL generate a distinct follow-up email for each application — two applications to the same company SHALL produce different email content reflecting the specific role context of each.
5. WHEN Company_Intel is not available for the target company, THE Follow_Up_Agent SHALL fall back to generating a personalized email using only the job description and resume context, without using generic template text.
6. WHEN a follow-up email is generated, THE System SHALL present the draft to the user for review and approval before sending, enforcing the HITL_Gate.
7. WHEN a follow-up email is sent after user approval, THE Follow_Up_Agent SHALL log the send event to the `agent_runs` table with `agent_type = 'followup'` and update the `job_applications.followup_day5` or `followup_day12` timestamp accordingly.
8. THE Follow_Up_Agent SHALL route all LLM calls through the Model_Router using the user's active `user_model_settings`.

### Requirement 8: Resume Score & ATS Analyzer

**User Story:** As a job seeker, I want my tailored resume scored against the job description for ATS keyword coverage, readability, and format compliance before I apply, so that I can fix issues that would cause automatic rejection.

#### Acceptance Criteria

1. WHEN a user requests an ATS analysis for a resume and job description pair, THE System SHALL compute an ATS_Score between 0 and 100 composed of three weighted sub-scores: keyword coverage (50%), readability (30%), and format compliance (20%).
2. THE System SHALL identify and return a list of keywords present in the job description but absent from the resume, ordered by estimated ATS importance (title matches first, then required skills, then preferred skills).
3. THE System SHALL return a readability assessment including: Flesch-Kincaid reading level, average sentence length, and a pass/fail flag for each of the following format compliance checks: presence of contact information, absence of tables and graphics, standard section headings, and file format compatibility.
4. WHEN the ATS_Score is below 60, THE System SHALL return at least three specific, actionable improvement suggestions referencing the exact resume section and the missing keyword or format issue.
5. THE System SHALL complete ATS analysis and return results within 15 seconds of receiving the resume text and job description text.
6. WHEN an ATS analysis is completed, THE System SHALL store the result in a new `ats_scores` table associated with the `user_documents` resume record and the `job_applications` record (if provided), with a `scored_at` timestamp.
7. THE System SHALL expose ATS analysis via a new API route at `POST /api/v1/resume/ats-score`.
8. WHEN a user views the ATS score results, THE System SHALL display the overall score, the three sub-scores, the missing keyword list, and the improvement suggestions in a structured breakdown UI.
9. THE System SHALL allow ATS analysis to be triggered both standalone (from `/resume/optimize`) and inline within the job application flow (from `/jobs/[id]`).
10. IF the resume text or job description text submitted for analysis is fewer than 100 characters, THEN THE System SHALL return an error response with HTTP status 422 indicating that the input is too short for meaningful analysis.

### Requirement 9: Multi-Resume Profile Manager

**User Story:** As a job seeker, I want to maintain multiple named resume personas and have the system automatically select the best-fit persona when applying to a job, so that I can target different roles without manually switching resumes.

#### Acceptance Criteria

1. THE System SHALL allow a user to create, name, update, and delete Resume_Personas, with each persona linked to a primary resume document in `user_documents` and a set of target role keywords.
2. THE System SHALL enforce a maximum of 10 active Resume_Personas per user account.
3. WHEN a user applies to a job, THE System SHALL automatically select the best-fit Resume_Persona by computing a keyword overlap score between the job description and each persona's target role keywords, and selecting the persona with the highest score.
4. WHEN the automatic persona selection runs, THE System SHALL display the selected persona name and its match score to the user before proceeding, allowing the user to override the selection.
5. IF no Resume_Persona achieves a keyword overlap score above 20%, THEN THE System SHALL notify the user that no strong persona match was found and prompt the user to manually select a persona or create a new one.
6. WHEN a Resume_Persona is selected for an application, THE Resume_Agent SHALL use the persona's linked primary resume document as the base for RAG-powered tailoring.
7. THE System SHALL store Resume_Persona definitions in a new `resume_personas` table with fields: `id`, `user_id`, `name`, `description`, `primary_resume_id` (FK to `user_documents`), `target_keywords` (text array), `created_at`, `updated_at`.
8. THE System SHALL expose Resume_Persona management via new API routes: `GET /api/v1/resume/personas`, `POST /api/v1/resume/personas`, `PUT /api/v1/resume/personas/{persona_id}`, and `DELETE /api/v1/resume/personas/{persona_id}`.
9. WHEN a Resume_Persona's primary resume document is deleted from `user_documents`, THE System SHALL set the persona's `primary_resume_id` to null and display a warning to the user that the persona requires a new primary resume.
10. THE System SHALL display all Resume_Personas on the `/resume/versions` page with their name, description, linked resume filename, and target keywords.

### Requirement 10: LinkedIn Connection Outreach Automation

**User Story:** As a job seeker, I want the platform to identify relevant recruiters and hiring managers at my target companies, draft personalized connection request messages, and queue them for my approval before sending, so that I can build my network efficiently without spending hours on manual outreach.

#### Acceptance Criteria

1. WHEN a user triggers LinkedIn outreach for a target company, THE System SHALL use the Proxycurl_Service to identify recruiters and hiring managers at that company, returning a list of up to 10 relevant contacts with name, title, LinkedIn URL, and company.
2. THE System SHALL filter identified contacts to include only those with titles containing recruiter, talent acquisition, hiring manager, engineering manager, or director-level keywords, excluding individual contributor roles.
3. WHEN a list of contacts is identified, THE System SHALL draft a personalized LinkedIn connection request message for each contact, incorporating the contact's name, title, the user's relevant experience from their active Resume_Persona, and Company_Intel if available.
4. THE System SHALL enforce a maximum connection request message length of 300 characters, as required by LinkedIn's platform constraints.
5. WHEN connection request messages are drafted, THE System SHALL queue all messages in a new `linkedin_outreach_queue` table with status `pending_approval` and present them to the user for review before any message is sent.
6. THE System SHALL enforce the HITL_Gate: no LinkedIn connection request message SHALL be sent without explicit per-message user approval.
7. WHEN a user approves a connection request message, THE System SHALL update the message status to `approved` and log it to the `agent_runs` table with `agent_type = 'linkedin_outreach'`; the actual send action SHALL be executed via PinchTab only after approval.
8. WHEN a user rejects or edits a connection request message, THE System SHALL update the message in the `linkedin_outreach_queue` table and allow the user to re-submit for approval without re-running the full identification step.
9. THE System SHALL expose LinkedIn outreach functionality via new API routes: `POST /api/v1/linkedin/outreach/identify`, `GET /api/v1/linkedin/outreach/queue`, and `POST /api/v1/linkedin/outreach/{message_id}/approve`.
10. IF the Proxycurl_Service returns no contacts for the target company, THEN THE System SHALL return an informational response with HTTP status 200 and a message indicating that no matching contacts were found, without creating any queue entries.
11. THE System SHALL route all LLM calls for message drafting through the Model_Router using the user's active `user_model_settings`.
12. WHEN PinchTab executes a LinkedIn connection request send, THE System SHALL use human-like delays between actions as required by the platform's automation constraints, with a minimum delay of 3 seconds between consecutive send actions.
